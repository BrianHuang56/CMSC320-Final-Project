Predicting Likelihood of Winning Based On Agent in Valorant

Valorant is a free to play first person tactical hero shooter game made by Riot Games. Initially being teased as Project A, Valorant was released on June 2nd, 2020 and now has an average of 15 million players per month. Valorant copies its core gameplay from another tactical shooter, Counter Strike. Each team consists of five agents, each having unique abilities that serve their own utility. One team is called the "Defenders", and the other team is the "Attackers". The attackers spawn with an object called the spike that they want to detonate at certain locations on the map, while the defenders want to prevent them from detonating the spike at those locations. Each round has a timer, being 100 seconds long. If time rounds out before the attackers can plant the spike, then the defenders win. If the attackers plant the spike, then the defenders have 45 seconds to defuse the spike before it detonates. The game consists of two halves, each half consisting of 12 rounds. At the end of the half, the teams will switch roles and continue to play until one team reaches 13 rounds won. At the time of writing this, there are only 7 maps and 19 agents. More information can be found on their website here.

Riot Games runs competitive circuits within regions and then the teams that prove themselves to be the best are then allowed to compete in the biggest international Valorant event. In these events, we see diversity in team comps throughout different maps and regions. This begs the question, what agents are the best, and what statistics can we use to say an agent is good? Is there a way to objectively determine whether or not an agent is good? This tutorial aims to try to answer that question.

Data Collection/Curation

We will use Python 3 with many imported libraries like requests, BeautifulSoup, seaborn, and sklearn to name a few. In addition, we will be requesting from the match webpages from 2 tournaments, VCT Stage 1: Masters Reykjavik, the most recent international event as of writing this, and VCT North America Stage 1: Challengers. The reason behind choosing these two is that they are both on the same patch, meaning the versions of the game are the same. Additionally, the reason behind choosing VCT North America Stage 1: Challengers is that North America did end up winning VCT Stage 1: Masters Reykjavik, meaning that we would be getting more data from the best team in the world at the time. As to why there are 3 links for 2 tournaments, the match pages on the website we are scraping from, vlr.gg, are separated by the stage of the event. To get more data points, I scraped from both the group stage and the main event. Any team that makes it to the group stage is going to be extremely strong, meaning that they will be a good team to take data from.

Procuring Data via Scraping¶

Since we have requested the match pages from the events, now we have to extract the match pages from each webpage. The way we will do that is utilizing BeautifulSoup and regexes. We will first parse each webpage using BeautifulSoup. I chose to parse it with html.parser because I thought it was the easiest to work with out of all of them. After we parse the webpage, we will then create an array that will hold all of the match webpages and create a regex that checks if the href link starts with 5 digits. The reasoning behind this is that every href link that starts with 5 digits is a match page and there are many href links within the webpage itself. We will now go through every webpage and check every "a" attribute, since every href will be linked the an a attribute, and check if it starts with 5 digits. If it does, then we append it to the original website link and add it to our matches array.

Now that we have every single match webpage, now we have to get the individual game webpages. The way we will do this is by parsing each match page using BeautifulSoup, as we did before for each match page, and find each div attribute that is related to the buttons to switch to each game in order to find the data-game-id. This number is used in the urls to go to each specific game. Once we find each one, then we append it to our match link along with "/?game=", simply because that's how the urls were formatted, and add it to our games array.

#Getting all the individual game links from all the match webpages
games = []
for match in matches:
    req = requests.get(match)
    soup = BeautifulSoup(req.content, "html.parser")

Now comes the hard part, getting all of the statistics for each game. The website is formatted where each team in the game has their own table, and hold a bunch of statistics for each player, such as their kills, deaths, assists, and more. We will hold these statistics in tables, each serving their own purpose. We will now go through each game, utilizing BeautifulSoup to request and parse the webpage, and look for specific attributes. The first thing we must find is what game is currently active. The website has buttons that change what is currently displaying, meaning that we can accidentally access a table for a different game, which is not what we want. The reason as to why we are accessing these games individually if all the tables are accessible on one page is that vlr.gg will also hold as many "vm-stats-game" attributes as there are potential games, meaning that you can go into one where there is zero data to be gathered and can be very annoying to deal with. In order to find the one we are trying to look at, the attribute will not be "vm-stats-game", but instead "vm-stats-game mod-active". Once we get into the active game, then we can check the scores by going into the header and into the team class and checking the scores. If the "team" class has the "score mod-win" class, that means the first table holds the winning team. If the "team mod-right" class has the "score mod-win" class, then the second table holds the winning team. In order to differentiate between the two teams, we will use a variable to tell us if the agent we are currently looking at is on the winning team or not. After this, we will get the map by finding the span attribute with a specific style. In the same spot where we checked which team won, we can also find the amount of rounds each team has. We can then add them together to get the total number of rounds each game lasted. This will be important later. We can also get the map name, but the text we get from the attribute will say "mapnamePICK" most of the time. The times where it does not is if is the last possible game of the match, since the decider is picked by elimination. So we will check if the text has "PICK" in it by checking the length and if it does, removing it. Now comes the hardest part, getting information out of the tables. The reason behind why this is hard is that many of the attributes have the same name, making it hard to differentiate between certain statistics and not getting statistics we are not interested in. For instance, 4 statistics share the same class, "mod-stat". The 4 statistics are ACS (Average Combat Score), KAST (Kill, Assist, Trade, Survive Percentage), ADR (Average Damage per Round), and headshot percentage. Thankfully, ACS is the first statistic, making it easy to get, and ADR has unique span class value, making it easy to get as well. The rest of the statistics we are interested in each have their own unique class name, making them also easy to get. However, in order to get the agent the player was playing, we will have to do a little more than just getting the text from attribute. Instead of having the agent in text, they have a picture of the agent, so we have to find the title of the picture in order to get the agent. Throughout this process for each agent, we are adding each agent to the agent_used array. The reason for this array is to check if there are mirror match ups for agents. Valorant is not like their other game, League of Legends, where each agent in the game is unique and there cannot be mirror matchups. Since this is the case, I decided that, in order to have a win rate statistic that has actual meaning, if there was an agent mirror match up, the amount of games won by that agent would increase by two instead of one. The reason behind this is that in the end I will divide the amount of games won by that agent on that map by the amount of times they were picked on that map. Otherwise, if an agent was picked 100% of the time, their win rate would be 50%, which seems contradictory in terms of how their perceived strength is on that map. It would make more sense for their win rate to be 100% instead of 50%. After all the scraping is done, we then calculate certain statistics based on other statistics we just got, such as KDA, which is (Total Kills + Total Assists) / Total Deaths, pick rate of each agent, win rate of each agent, overall ADR and overall ACS.

Data management/representation

I have made many tables to use to store the data we get from the games, each serving their own purpose. We have a win rate table, which will hold the win rates of each agent on each map and overall. As you might notice, there are many rows in the win rate table that are NaN. This is because the pick rate for those agents on those maps are 0%. If we were to try to calculate the win rate of those agents, it would quite literally be undefined. So we are unable to put a value on their win rate. And we cannot assume a value, since pick rate is not necessarily indicative of win rate, as we can see from Yoru, who has an extremely low pick rate but has a decent win rate on certain maps. Assigning them a value of 0% would extremely skew the data, especially since the agent Phoenix was literally never picked for any map. So we will leave the data as NaN to represent undefined. We also have a pick rate table, which shows the pick rate of agents on each map and overall. There is also an agent stats table, holding many statistics for each agent, such as KDA, ADR, ACS, total games played, and more. Surprisingly, Yoru, one of the lowest picked agents, has the best stats out of all of the other agents. There is also a team game information table, which holds the information of a teams composition, what map they played, and whether or not they won. And the last table is very similar, but instead for being for the whole team, it is for each specific agent and their statistics from that specific game, such as their kills, deaths, ADR, etc.

Insights Attained

Valorant is a game with an extreme amount of possibilities and combinations, summing up to 11,628 possible team compositions. Understanding what agents are the best is vital to increasing a team's chance of winning. While this tutorial has tried to find the "best agents", Valorant is a game that is much more complex. Certain agents are going to synergize better with each other based on what abilities and utility they have, and each agent play a certain role. For instance, while Jett and Raze both have quite a high win rate and pick rate, very rarely would a team actually play both agents at the same time. This is because they overlap in their role and the marginal value you get from playing both of them is lower than playing one of the two and playing another supporting agent, such as Sova. There is only so much statistics can do when looking at an unsolvable game, so thinking rationally about what agents mesh well together and what the map calls for is the most important. Having played for the UMD Valorant Team for four seasons now, Valorant is a game that is so unexplored and has much more to be discovered. Each region in the world usually has vastly varying team comps, each playing to their players strengths. It will not be surprising to see a few months from now that the statistics look much different. Additionally, there is still so much to be explored. As I stated earlier, each region has varying team comps for each map. For example, Paper Rex, at team from the Asian Pacific, run Yoru and Breach on Bind, something that no other team at VCT Masters 1: Iceland Reykjavik did. And they won two of the three games they played on Bind. This tutorial is just a small portion of the full picture, there are many regions that still need to be explored.
